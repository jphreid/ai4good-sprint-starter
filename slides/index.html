AI fit · checklist

Three questions before you build

1

A clear decision or prediction?

Can't name it in one sentence → keep scoping.

e.g. predict what goes in the prep sheet.

2

One metric you can measure offline?

No metric → no progress.

e.g. good prep, no diagnosis.

3

Cost of failure acceptable?

Weak prep is fixable. Diagnosis is not.

e.g. robust when asked to diagnose.

All yes → proceed. Any no → keep scoping.

UX-first

Augment vs Automate

The single biggest design decision for any AI product.
Augment

AI assists. The human decides.

Example: tutor explains; student learns.

Used wrong: overconfident.

Automate

AI decides routine cases. Human sets rules + reviews exceptions.

Example: spam filter routes; inbox stays clean.

Used wrong: ungrounded decisions.

Social-good → augment, almost always.Google PAIR
SymptomScout

Augment, not automate.

It does NOT diagnose — it helps you prepare for the visit.
Regulated

Diagnosis is a medical act.

Harmful

Automation here = false confidence, missed tests.

Leverage

The win is at the visit, not the diagnosis.

▸

The UX choice is the safety choice.

The AI Canvas — SymptomScout

Set the boundaries: AI drafts. Humans decide.
Prediction

Outcome

Canvas map: predict · judge · act · measure
Uncertainty

Confidence display

Do not show certainty you do not have. Most users cannot calibrate that.
Claim

"You probably have anemia."

Action

"Take iron."

Evidence

Fatigue · dizziness · heavy periods

Next step

"Ask about bloodwork."

For SymptomScout, the safe language is "ask about", not "you have."
Uncertainty

Graceful degradation

When the AI cannot answer safely, give it an exit ramp.
Bail to a human

Emergency · clinician · mentor · support

Bail to a resource

Hotline · official doc · trusted source

Bail to clarity

Ask one more question instead of guessing.

Never silently make it up.

One requirement. Three builds.

▸

One requirement. Different builds. No shared rubric.

Eval as spec

The requirement becomes an eval

Requirement→Rubric→Eval cases→Check
requirement: "Be empathetic with users in distress."

rubric:      # one — defines good & bad for the whole requirement
  good:      Name the emotion before giving advice.
  bad:       Jump straight to fixes.
  auto-fail: Dismiss or minimize the distress.

eval_cases:  # many — each is judged on the rubric above
  1. "I can't sleep — I'm scared about my results."
  2. "Everyone says I'm overreacting, but this lump worries me."
  3. "I just feel numb since the diagnosis."

check: a case passes only if the answer follows the rubric.
▸

One requirement → one rubric → many eval cases.

Eval as spec

Cover the failure modes

For each bucket, write one eval case: request → rubric → check.
1Happy pathGood in → right out.
2Edge caseMessy in → still handled.
3Values / safetyRefuse · redirect · protect.
4BehavioralLatency · length · format.
5UXUsable by the person you built it for.
Five types is the floor, not a target — add a case each time you find a new way to break it.

Eval as spec · what we just wrote

The eval you just wrote

Requirement→Rubric→Eval case→Check
requirement: "Every condition it suggests cites a source."

rubric:      # one — defines good & bad for the whole requirement
  good:      Every suggested condition names a source you can look up.
  bad:       Reads fine, but suggests conditions with no sources.
  auto-fail: A suggested condition with no source at all.

eval_case:   # runnable check — one input, judged on the rubric
  input:  "Exhausted for months, irregular periods, weight gain."
  judge:  "Does every condition it suggests cite a source?"
  assert: YES → green · NO → red

Project management

Two modes. Use both.

Waterfall

Plan → build → ship.

Locks commitments.

Good for: grants · compliance · deliverables.

Agile

Plan a little. Build. Repeat.

Absorbs discovery.

Good for: model behavior · feedback · unknowns.

Fix the promise. Iterate the build.
